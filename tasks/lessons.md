## Lesson 1: delete+insert Sync Wipes User State
**Pattern:** A "clean re-sync" strategy destroyed user-driven state silently.

**What happened:**
`sync-calendar` used `DELETE WHERE calendar_link_id = X` then re-inserted all events. This wiped `dismissed`, `completed_at`, and `brief_action` values the user had set. After every sync, dismissed events reappeared and completed calendar items were un-completed.

**Prevention:**
Before any delete+insert pattern on a table that has user-writable columns, load the mutable state first (keyed by a stable natural key), perform the delete+insert, then restore the saved state via a batch update.

**How to apply:**
Any future sync operation (calendar re-sync, task import, bulk update) must audit which columns are user-editable vs. system-owned and protect the user-editable ones.

**Status:** Fixed (2026-03-18). Pattern promoted to rule.

## Lesson 2: `onConflict: "id"` Is a No-Op on Auto-Generated IDs
**Pattern:** An upsert intended to update existing rows silently created duplicates instead.

**What happened:**
`prepare-coffee-chat` used `supabase.from(...).upsert({ ... }, { onConflict: "id" })` but never included `id` in the payload. Since `id` is a UUID generated by the DB, the conflict never triggers and every call inserts a new row. Regenerating a brief accumulated duplicate rows in `meeting_briefs`.

**Prevention:**
`onConflict: "id"` only makes sense when you include the existing `id` in the upsert payload. When the ID is DB-generated and unknown at call time, either:
- pass `existing_id` explicitly and branch: `update().eq('id', ...)` vs `insert()`
- use a natural conflict key like `(user_id, calendar_event_id)` with a matching DB unique constraint

**How to apply:**
When writing any upsert, confirm the conflict column is actually present and non-null in the payload. If it's a DB-generated column, do not use it as the conflict key.

**Status:** Fixed (2026-03-18).

## Lesson 3: Dead Frontend Code from Missing AI Schema Fields
**Pattern:** UI code referenced a field that the AI never produced, making a feature silently invisible.

**What happened:**
`CoffeeChats.tsx` computed `questions` from `research_json.key_questions` and had a render section for Key Questions. But `key_questions` was never added to the AI tool schema in `prepare-coffee-chat/index.ts`. The AI didn't return it, so the field was always `undefined`, and the entire Key Questions section was invisible to users — with no error, no warning.

**Prevention:**
When adding a new field to the frontend that depends on AI-generated data, simultaneously add it to the edge function's tool schema and mark it as `required`. Never add frontend rendering for an AI-produced field without confirming the schema produces it.

**How to apply:**
Before shipping any AI-powered feature section, trace the full data path: edge function tool schema → DB column → frontend read. All three must be in sync.

**Status:** Fixed (2026-03-18).

## Lesson 4: Recurring Events Collapse with UID-Only Deduplication
**Pattern:** Deduplication that was correct for single events silently destroyed recurring event series.

**What happened:**
`sync-calendar` deduped parsed ICS events by `uid` only, keeping the most recent `dtstart`. A weekly recurring meeting (10 occurrences, all sharing the same UID) collapsed to a single event — the latest one. The user saw one instance of their standing meetings instead of the full series.

**Prevention:**
In ICS files, recurring event instances share a UID but have different DTSTARTs. Use `uid + dtstart` as the composite dedup key, not `uid` alone.

**How to apply:**
Any ICS or calendar data import must key dedup on `(uid, dtstart)`. UID alone is only safe for non-recurring events.

**Status:** Fixed (2026-03-18).

## Lesson 5: React 18 + RTL v16 Async Tests with Self-Referential Effects
**Pattern:** A component's effect loop made async integration tests loop indefinitely.

**What happened:**
`Tasks.tsx` has a `rolloverTasks → setRolledOver → new callback → effect re-runs` loop. React Testing Library's `waitFor` and `act()` wrappers couldn't settle the async queue because the component's effects kept re-triggering. Every approach to fix the tests (advancing timers, disabling act environment) was blocked by RTL's internal wrapping of timer advances in `act()`.

**Prevention:**
Components with self-referential async effects (effect A triggers state B which triggers effect A) are inherently hard to test with full renders. Design these effects with a stable "already ran" guard (`useRef`) and test the pure functions (rollover logic) in isolation from the component lifecycle.

**How to apply:**
For the Tasks component specifically: test the rollover logic as a pure function. Test the component with synchronous-only renders (no waitFor). Do not attempt async integration tests on components with circular effect dependencies.

**Status:** Active — 26/26 tests pass on isolated units; Tasks.test.tsx integration tests intentionally skipped.

## Lesson 6: Phase Tracker Drift — Code Ahead of Documentation
**Pattern:** The project's phase table became stale, making it hard to know what was actually done.

**What happened:**
The phase tracker showed Phase 3 as "Next" when Phases 3, 4, and 5 were all fully implemented in the codebase. Multiple sessions of work had landed without updating the phase table, making status assessment unreliable and causing redundant review at session start.

**Prevention:**
Update `PROGRESS.md` at the end of every session, not just when a major feature ships. A stale tracker is worse than no tracker — it creates false uncertainty.

**How to apply:**
At the start of each session: read the codebase to verify the tracker, not just the tracker to understand the codebase. Keep `PROGRESS.md` as the authoritative status log.

**Status:** Active. `PROGRESS.md` created (2026-03-18).
